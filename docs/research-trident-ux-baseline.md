## Fuzz target UX baseline (from `examples/hello_world/`)

User writes 3 files in `trident-tests/fuzz_0/`:
- `Trident.toml` — config
- `fuzz_accounts.rs` — account address storage
- `types.rs` — auto-generated instruction bindings
- `test_fuzz.rs` — actual fuzz logic

### test_fuzz.rs pattern

```rust

#[derive(FuzzTestMethods)]
struct FuzzTest {
    trident: Trident,
    fuzz_accounts: AccountAddresses,
}

#[flow_executor]
impl FuzzTest {
    fn new() -> Self { ... }

    #[init]
    fn start(&mut self) {
        // Setup: load external programs, derive PDAs, airdrop
        let account = self.trident.get_account(&pubkey!("..."));
        invariant!(account.executable());
        // ...
        let res = self.trident.process_transaction(&[ix], Some("Initialize"));

    // #[flow] methods would go here — drive multi-tx sequences

```

## Observations

### Strengths
- **Macro-driven lifecycle** (`#[init]`, `#[flow]`, `#[flow_executor]`) gives clear structure
- **Auto-generated instruction bindings** from Anchor IDL = users don't hand-write borsh
- **Built-in cheatcodes**: `forward_in_time`, `random_from_range`, `airdrop`, `get_account`, `process_transaction`
- **invariant! / invariant_eq!** macros for assertions
- **PdaSeeds helper** for clean PDA derivation in account setup

### Weaknesses (= solinv wedges)
1. **All invariants manually written.** No auto-detection of signer-skip/owner-skip/discriminator-skip/PDA-forge. User must know to write each check.
2. **Anchor-centric.** Examples are all Anchor programs (hello_world, cpi, metaplex, token). Pinocchio / native program support is unclear — `trident-spec.json` suggests Anchor IDL is the primary input format.
3. **No mainnet corpus seeding** — random byte mutation only. No "fuzz with realistic state" path visible.
4. **No disclosure formatter** — bug found → user manually writes Immunefi submission.
5. **Manual external program loading** — `start()` body loads Drift / Jupiter programs by pubkey, no auto-import.
6. **types.rs is generated** but **DO NOT EDIT** warning means users can't add invariants there. All invariant logic ends up in test_fuzz.rs which gets unwieldy at scale.

### What solinv provides (vs Trident UX)

| Capability | Trident | solinv |
|---|---|---|
| Invariant auto-detection (15-20 species) | ✗ manual | ✓ via `solinv::invariants::*` library |
| Mainnet corpus seeding | ✗ | ✓ via `solinv-corpus` (Yellowstone) |
| Disclosure formatter | ✗ | ✓ `solinv disclose` → Immunefi/Sherlock |
| Pinocchio first-class | ✗ Anchor-centric | ✓ explicitly supported |
| Engine | TridentSVM (custom) | Crucible (LibAFL+LiteSVM) |
| Coverage-guided | partial (sbf-instrumentation branch) | inherited from Crucible |

## Decision: don't compete with Trident's macro UX

Trident's `#[flow]` macro architecture is opinionated and well-developed. Crucible's `#[fuzz_fixture]` is competing in similar territory. **solinv should NOT design a third macro vocabulary.**

Instead solinv exposes its invariants as **library functions callable from either Crucible's `#[invariant_test]` or (someday) Trident's `#[flow]`**. Users adopt whichever macro framework they prefer; solinv adds the auto-detection + corpus + disclosure value on top.

This is consistent with the strategic choice to run as Crucible plugin (Pattern A from research-crucible-integration.md).
